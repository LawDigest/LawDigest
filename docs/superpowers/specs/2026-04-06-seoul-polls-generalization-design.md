# 서울시 여론조사 확장 및 광역시·도 일반화 설계

작성일: 2026-04-06

## 1. 목표

현재 여론조사 수집/파싱 파이프라인은 `제9회 전국동시지방선거 경기도 광역단체장선거`를 중심으로 정리되어 있다. 이번 작업의 목표는 다음 두 가지를 동시에 달성하는 것이다.

1. 서울특별시장 선거 여론조사를 실제로 수집하고 PDF 파싱 및 픽스처 검증까지 완료한다.
2. 경기도 전용에 가까운 현재 타겟 구조를 광역시·도 공통 구조로 일반화해 이후 다른 시·도 확장이 가능하도록 만든다.

이번 작업은 운영 중인 안정 버전을 보존하는 성격이 아니므로, 하위 호환을 유지하지 않고 새 구조로 바로 전환한다.

## 2. 범위

포함 범위:

- 수집 대상 정의 모델 개편
- NESDC 검색 파라미터 생성 및 타겟 매칭 일반화
- 서울시 타겟 추가
- 서울시 실제 PDF 수집 및 파서 적용
- 필요한 기관 파서 보완
- 서울 픽스처 및 회귀 테스트 추가
- 관련 문서 및 스크립트 경로/설명 정리

제외 범위:

- 웹 UI 또는 운영 배포 반영
- 전국 단위 전체 확장
- 교육감/기초단체장 등 다른 선거 종류 동시 확장

## 3. 설계 원칙

- 서버 사이드 필터링(`poll_gubuncd`, `search_cnd`, `search_wrd`)은 유지한다.
- 지역 확장은 `타겟/수집 레이어`에서 해결하고, `파서 레이어`는 기관 포맷 중심으로 최소 수정한다.
- 설정은 사람이 읽기 쉬운 선언형 구조를 유지하되, 지역/선거/타겟 책임을 분리한다.
- 하위 호환 코드는 두지 않는다. 기존 경기 전용 구조는 새 구조로 치환한다.

## 4. 목표 구조

### 4.1 모델 계층

기존 `PollTarget`은 검색 파라미터, 지역 필터, 선거 필터가 한 객체에 섞여 있다. 이를 다음 계층으로 분리한다.

- `RegionSpec`
  - 예: 서울특별시, 경기도
  - NESDC 검색어(`search_wrd`)와 내부 지역 매칭 기준(`region`)을 가진다.
- `ElectionSpec`
  - 예: 제9회 전국동시지방선거 / 광역단체장선거
  - `poll_gubuncd`, `election_type`, `election_names`를 가진다.
- `PollTarget`
  - 실제 실행 대상
  - `slug`, `region_key`, `election_key`, 기관 제한, 예외 PDF 목록 등을 가진다.

런타임에서는 `PollTarget`이 `RegionSpec`과 `ElectionSpec`을 조합한 완성형 검색/필터 정보를 제공하도록 구성한다.

### 4.2 설정 파일

기존 `poll_targets.json`을 다음과 같은 3섹션 구조로 재편한다.

- `regions`
- `elections`
- `targets`

예시 개념:

- `regions.seoul`: `search_wrd=서울특별시`, `region=서울특별시 전체`
- `regions.gyeonggi`: `search_wrd=경기도`, `region=경기도 전체`
- `elections.local_9th_governor`: `poll_gubuncd=VT026`, `election_type=제9회 전국동시지방선거`, `election_names=["광역단체장선거"]`
- `targets.seoul_mayor_9th`: `region_key=seoul`, `election_key=local_9th_governor`
- `targets.gyeonggi_governor_9th`: `region_key=gyeonggi`, `election_key=local_9th_governor`

이 구조로 바꾸면 같은 선거 정의를 여러 지역이 공유할 수 있다.

## 5. 코드 변경 방향

### 5.1 `targets.py`

- `RegionSpec`, `ElectionSpec`, 개편된 `PollTarget` 도입
- 새 JSON 구조 로더 구현
- `matches_target()`이 완성형 타겟 기준으로 지역/선거/기관 조건을 검사하도록 수정
- `parse_title_region()`은 유지하되, 테스트를 보강해 서울/경기 모두 안정적으로 해석되는지 검증

### 5.2 수집기(`crawler.py`, 관련 스크립트)

- `crawl_for_targets()`는 현재처럼 `(poll_gubuncd, search_cnd, search_wrd)` 조합 단위로 묶되, 이 값을 새 타겟 모델에서 가져오도록 변경
- `collect_poll_list.py`, `download_pdfs.py`, `check_pdfs.py`, `parse_all.py`, `probe_parsers.py`는 계속 `slug` 중심으로 실행
- 출력 경로 및 로그 문구에서 경기 전용 표현 제거

### 5.3 파서 레이어

- 파서 선택 기준은 지역이 아니라 조사기관 포맷이므로 기본 구조는 유지
- 서울 PDF를 실제로 수집해 기존 파서로 적용해 보고, 실패하는 기관만 최소 수정
- 신규 기관이 발견되면 해당 기관 파서를 추가하고 레지스트리에 등록

## 6. 데이터 및 테스트 전략

### 6.1 수집 검증

- 서울 타겟이 올바른 NESDC 파라미터로 검색되는지 검증
- 경기/서울 타겟 결과가 서로 섞이지 않는지 검증
- slug 단위 수집 스크립트가 새 구조에서도 그대로 동작하는지 확인

### 6.2 파서 검증

- 서울 PDF 샘플을 실제로 다운로드해 파서 적용
- 파싱 성공 케이스는 픽스처 JSON 생성
- 파싱 실패 케이스는 기관별 원인을 분류해 보완

### 6.3 회귀 테스트

- 기존 경기 픽스처 테스트 유지
- 서울 픽스처 추가
- 타겟 로딩/매칭 단위 테스트 추가

완료 기준:

- 경기 기존 테스트가 계속 통과한다.
- 서울 대표 PDF들이 픽스처 기반 검증을 통과한다.
- 새 설정 구조로 `slug` 기반 수집 스크립트가 동작한다.

## 7. 리스크

### 7.1 서울 PDF의 신규 기관 포맷

서울 데이터에 경기에서 없던 조사기관 포맷이 섞여 있을 수 있다. 이 경우 일반화 작업보다 신규 파서 개발이 일정의 핵심 변수가 된다.

대응:

- 목록 수집 후 pollster 분포를 먼저 확인한다.
- 기존 기관부터 우선 처리하고, 신규 기관은 우선순위를 나눠 대응한다.

### 7.2 경기 전용 가정의 잔존

문서, 경로명, 테스트 설명, 스크립트 usage 예시 등에 경기 전용 표현이 남아 있을 가능성이 높다.

대응:

- 설정 구조 교체와 함께 검색어/지역명 하드코딩을 정리한다.
- 서울/경기 둘 다 사용하는 테스트나 예시로 교체한다.

## 8. 구현 순서

1. 타겟 설정 구조와 모델 계층 개편
2. 타겟 로더/매칭 테스트 작성
3. 수집 스크립트와 크롤러를 새 구조로 전환
4. 서울 타겟 추가 후 목록 수집
5. 서울 PDF 다운로드 및 조사기관 분포 확인
6. 기존 파서 재사용 검증
7. 필요한 기관 파서 수정 또는 신규 추가
8. 서울 픽스처 생성 및 통합 테스트 보강
9. 문서/스크립트 설명 정리

## 9. 비목표

- 이번 단계에서 전국 모든 광역시·도를 한 번에 추가하지 않는다.
- UI/시각화 계층까지 한 번에 연결하지 않는다.
- 구 구조와 신 구조를 동시에 지원하지 않는다.
